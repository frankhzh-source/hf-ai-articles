# LLM-native 清洗方案 — 传统 vs 现代对比

> 核心洞察：正则+规则+标注平台这套"老四样"，现在 LLM 一条 prompt 就覆盖了。

---

## 一、传统方案的问题在哪

我前面写的6步SOP，本质是**把人脑的清洗逻辑翻译成机器规则**：

```
原始对话 → 去敏规则 → 去噪规则 → 意图关键词 → QA抽取规则 → 格式化
           ↑ 写正则     ↑ 配黑词     ↑ 配词典     ↑ If-else     ↑ template
```

每个步骤都需要人工维护规则集，而且规则永远写不全——

```
规则能识别"有货吗"，但跟不上"那个还能拍不""链接还能下吗""补货了吗"
规则能识别"多少钱"，但跟不上"这个什么价""米数多少""到手多少"
规则能识别手机号，但追不上"VX 138xxxx""打我电话 138xxxxxxxx 不在线"
```

**这就是传统清洗最痛的维护成本**：不是第一次清洗难，是每次电商活动季，话术一变，规则就得跟着改。

## 二、LLM 方案：一条 prompt 搞定

```python
system_prompt = """
你是一个电商客服对话清洗专家。你的任务：
1. 脱敏：将手机号/微信号/地址替换为占位符
2. 去噪：去除自动回复/欢迎语/单字回复/系统消息
3. 打标：标记该段对话的核心意图（库存/尺码/价格/退换货/投诉等）
4. 抽取：从对话中抽取出高质量的问答对
5. 输出：严格按照以下JSON格式返回

输出格式：

    {"role": "buyer|seller", "content": "清洗后文本"}

  "intent": "主要意图",

  ],
  "notes": {
    "sensitive_found": ["脱敏项列表"],

"""
```

## 四、三档 LLM 清洗方案

### 方案1：一条 prompt 通吃（适合 < 10000条/月）

```
原始对话 → 单次 LLM 调用 → 结构化 JSON 输出
```

优点：最简单，零代码
缺点：长对话可能丢失细节，一次处理会话数有限

**推荐模型**：DeepSeek-V3（性价比最高）、Qwen-Max、豆包-Pro

### 方案2：多步 Agent 管道（适合 10000-50000条/月）

```
Step1 去噪Agent → Step2 打标Agent → Step3 QA抽取Agent → 合并输出
```

优点：每条任务更专精，支持并行处理
缺点：多一次调用，贵一点

**典型部署**：
```python
# Agent 1: 去敏去噪
agent_1 = lambda raw: llm(raw, prompt="脱敏+去噪")
# Agent 2: 意图打标
agent_2 = lambda clean: llm(clean, prompt="打标+情感")
# Agent 3: QA抽取+格式化
agent_3 = lambda tagged: llm(tagged, prompt="抽取QA对")
# 可并行跑
```

### 方案3：LLM + 少量代码兜底（推荐，平衡方案）

```
原始对话 → LLM 主清洗（90%对话走通）
               ↓ 失败时（格式异常/超长/特殊场景）
          代码兜底（正则 fallback）
               ↓
          质检Agent（LLM 自检输出质量）
```

**兜底场景**：
- 对话太长超出上下文 → 分段处理 + 合并
- 输出格式不符合 JSON → 重试一次 + 正则 fallback
- 纯图片/纯表情会话 → 跳过
- 多语混杂 → 标记后单独处理

**参考代码结构**：
```python
def llm_clean(dialogue, model="deepseek-v3"):
    """LLM主清洗，失败时 fallback"""
    try:
        result = call_llm(CLEAN_PROMPT + dialogue, model=model)
        parsed = json.loads(result)
        return parsed
    except (json.JSONDecodeError, Exception):
        # Fallback: 用简单的正则兜底
        return rule_based_fallback(dialogue)

def batch_clean(sessions, batch_size=20, model="deepseek-v3"):
    """批量处理，Batch API优惠"""
    results = []
    for batch in chunks(sessions, batch_size):
        # 并行调用 LLM Batch API
        batch_results = parallel_map(llm_clean, batch, executor=ThreadPool(4))
        results.extend(batch_results)
    return results
```

## 五、质量保障：LLM 自己质检自己

LLM 清洗最大的隐患是**幻觉**，所以一定要加一层自动校验：

```python
QCM_PROMPT = """
检查以下清洗结果的质量，逐项打分（1-5分）：
1. 脱敏完整性：是否有遗漏的手机号/地址/微信号？
2. 内容保真度：清洗后的意思是否与原文一致？
3. 意图准确性：标注的意图是否匹配对话内容？
4. QA相关性：抽取的问答对是否准确配对？

返回JSON：{"score": 总分, "issues": ["问题列表"], "pass": true/false}
"""
```

**质检策略**：

| 方法 | 实现 | 成本 | 覆盖率 |
|------|------|------|--------|
| 自动抽检 | LLM 对 10% 结果做 QC | 极低 | 10% |
| 自动全检 | 规则+LLM 异常检测 | 低 | 100% 异常 |
| 人工抽检 | 随机 200 条人工看 | 中等 | 0.5-1% |
| 自动修复 | 检测到问题后自动重跑 | 极低 | 仅修复异常 |

## 七、什么时候不用 LLM

虽然 LLM 能力很强，但仍有边界：

| 场景 | 为什么不用 | 替代方案 |
|------|-----------|---------|
| **数据不能出公司内网** | 敏感数据，不允许走外部 API | 用本地模型（Qwen-7B/DeepSeek-R1-Distill） |
| **实时清洗（毫秒级）** | LLM 延迟 > 500ms | 走规则闪电清洗 |
| **量极大（百万级/天）** | Token 成本会累积 | 先用规则粗洗，再用 LLM 精洗 |
| **纯结构化数据（字段固定）** | LLM 大材小用 | 规则+模板更快更准 |

## 八、结论

**你说得对。对于一个服装电商的客服对话清洗场景，最优解是：**

> 写一条高质量的 prompt，用 Batch API 批量跑，加一层 LLM 自检兜底。

就三条：
1. **一条 prompt** 替代 6 步 SOP + 规则集 + 标注平台
2. **Batch API** 月成本几十块，不是几千块
3. **LLM 自检** 替代大部分人工质检

传统那套规则+工具的清洗方式，在 LLM 面前已经是前朝遗物了。
