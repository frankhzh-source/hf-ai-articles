## 第一章：系统总览

### 1.1 系统架构图

```
┌─────────────────────────────────────────────────────────┐
│                   感知层（外部信号）                         │
│  WebSearch / 飞书订阅 / 行业RSS / 客户反馈入口            │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────────────┐
│               信号预处理 & 分类路由器                        │
│    热点相关 → 知识库  /  客户反馈 → 改进队列               │
│    异常信号 → 告警  /  竞品动态 → 策略评估                 │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────────────┐
│      反馈层（正+负信号 均采集，指向具体优化目标）              │
│  公众号数据 / LoRA训练 / 客户项目 / 内部运营               │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────────────┐
│            Elite 记忆系统（含TTL衰减机制）                    │
│  L1寄存器 ← L2向量库 ← L3冷存储 ← L4文件 ← L5压缩 ← L6图谱    │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────────────┐
│          SOP 流程引擎 ←——→ 标准验收体系                     │
│  怎么做（流程）              做成什么样算好（标准）            │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────────────┐
│              数据库层（持久化存储）                           │
│  SQLite + LanceDB + 文件系统                               │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────────────┐
│          正反馈循环 → 下一轮（实现复利优化）                   │
│        每次循环沉淀的数据，提升下一次的效率和质量              │
└──────────────────────────────────────────────────────────┘
```

### 1.2 核心原则

1. **数据主权第一**：所有自有数据不出本地，Elite系统离线运行
2. **人不可绕行**：校准节点（审核/验收）必须有人参与
3. **衰减必须存在**：未衰减的数据在12个月后自动降权
4. **负反馈是资产**：失败案例和成功案例同等重要
5. **反馈定目标**：每条反馈必须关联一个具体优化指标

### 1.3 角色定义

| 角色 | 职责 | 系统中对应 |
|------|------|-----------|
| 数据采集员 | 每日执行感知层任务 | AI自动化 |
| 信号分类器 | 将原始信号分类路由 | AI + 规则引擎 |
| 记忆管理员 | 管理Elite记忆系统健康度 | AI自动化 |
| SOP维护者 | 编写和维护操作流程 | 人（海风） |
| 标准制定者 | 制定和校准验收标准 | 人（海风） |
| 校准节点 | 定期人工校准循环方向 | 人（海风） |
| 系统监控员 | 监控系统运行状态 | AI自动化 |

## 第二章：感知层 — 外部信号捕获

### 2.1 信号源配置

#### 2.1.1 AI/科技热点（每日）

**执行人**：AI自动
**工具**：WebSearch
**执行时间**：每天上午 09:00

```
操作指令：
WebSearch(query="AI 人工智能 今日新闻 头条", topic="news")
WebSearch(query="大模型 最新发布 2026", topic="technology")
```

**输出格式**：
```
[日期] AI行业热点摘要
- 标题1 | 来源 | 核心要点 | 与我业务关联度（高/中/低）
- 标题2 | 来源 | 核心要点 | 与我业务关联度（高/中/低）
- ...
```

#### 2.1.2 行业动态（每周）

**执行人**：AI自动
**工具**：WebSearch
**执行时间**：每周一上午 09:30

```
操作指令：
WebSearch(query="AI落地 企业 实施 最佳实践 2026", topic="technology")
WebSearch(query="LoRA 训练 电商 AI 生图 最新进展 2026", topic="technology")
```

#### 2.1.3 客户反馈入口（按需）

**执行人**：人（海风）
**触发条件**：收到客户反馈/咨询/投诉时
**操作**：
```
1. 记录客户原始反馈内容
2. 标记反馈类型：需求/投诉/咨询/建议
3. 关联客户所属行业和项目
4. 将记录写入Elite记忆（L2层）
```

#### 2.1.4 竞品动态（每月）

**执行人**：AI自动
**执行时间**：每月1日
**工具**：WebSearch + 指定源抓取

### 2.2 信号预处理流程

#### 2.2.1 AI自动处理（每日）

```
Step 1: 采集原始信号（WebSearch）
Step 2: 去重（Elite L2检索，检查是否已有同类记录）
Step 3: 分类路由：
  └─ 热点相关  → 知识库候选
  └─ 客户反馈  → 改进队列
  └─ 异常信号  → 告警队列
  └─ 竞品动态  → 策略评估
Step 4: 输出信号摘要
Step 5: 写入Elite L2记忆（layer=signal）
```

#### 2.2.2 人工筛选（每日）

**执行人**：海风
**时间**：每天上午 10:00（建议5分钟）
**操作**：
```
1. 阅读AI输出的信号摘要
2. 标记"今日关注"（用于当日内容生产选题）
3. 标记"存档观察"（暂不处理，留做备选）
4. 标记"忽略"（与业务无关的噪音）
```

### 2.3 输出物规范

感知层每日输出两份产物：

**产物A：信号摘要（AI自动生成）**
```
文件位置：./感知层/YYYY-MM-DD_信号摘要.md
格式：Markdown
结构：头条新闻（3条）→ 行业动态（5条）→ 客户反馈（0-N条）
```

**产物B：关注标记（人标记后）**
```
格式：在信号摘要上直接批注
标记规则：[Focus] = 今日关注 | [Watch] = 存档观察 | [Skip] = 忽略
```

## 第三章：反馈层 — 信号采集与分类

### 3.1 反馈源清单

| 反馈源 | 采集方式 | 采集频率 | 优化目标指标 |
|--------|---------|---------|-------------|
| 公众号文章阅读量 | 截图提取/你手动报告 | 发布后1-2天 | 打开率 |
| 公众号完成率 | 截图提取/你手动报告 | 发布后1-2天 | 内容质量评分 |
| 公众号分享/留言 | 截图提取/你手动报告 | 发布后1-2天 | 共鸣度 |
| LoRA训练Loss曲线 | 日志文件读取 | 每轮训练后 | 效果评分 |
| LoRA生成效果 | 五维评分（一致性/细节/背景/跟随/材质） | 每轮测试后 | 综合得分 |
| 客户项目反馈 | 你主动记录 | 项目结束后 | 客户满意度 |
| 审核清单通过率 | 自动记录 | 每次生产后 | 流程健康度 |

### 3.2 反馈采集操作流程

#### 3.2.1 公众号数据采集

```
触发条件：文章发布后 24-48小时
执行人：你发截图/数据描述

操作步骤：
[你] "Day 166 阅读800 分享30 留言5 完成率63%"
[AI] 1. 解析数据
     2. 对比历史均值（Elite检索同类文章历史数据）
     3. 生成反馈记录
     4. 写入Elite L2记忆（layer=feedback, type=公众号）

反馈记录模板：
{
  "date": "2026-06-16",
  "article": "Day 166: 一个公众号IP白名单...",
  "metrics": {
    "reads": 800,
    "shares": 30,
    "comments": 5,
    "completion_rate": "63%"
  },
  "vs_history": {
    "reads_avg": 650,
    "shares_avg": 22,
    "comments_avg": 3
  },
  "insight": "完成率偏低（低于均值），标题吸引力足够但内容节奏需调整"
}
```

#### 3.2.2 LoRA训练反馈采集

```
触发条件：每轮训练结束后
执行人：AI自动

操作步骤：
Step 1: 读取训练日志中的Loss曲线数据
Step 2: 读取测试集五维评分结果
Step 3: 对比历史同类训练数据
Step 4: 生成反馈记录
Step 5: 写入Elite L2记忆（layer=feedback, type=lora_training）

反馈记录模板：
{
  "date": "2026-06-17",
  "sku": "黑色LOVE睡衣",
  "layer": "L2-人物三视图",
  "metrics": {
    "final_loss": 0.15,
    "convergence_speed": "1000steps",
    "test_score_avg": 4.2
  },
  "vs_history": {
    "prev_loss": 0.18,
    "prev_score": 3.9
  },
  "insight": "本次指标均优于前次，人物矩阵方案验证有效"
}
```

#### 3.2.3 客户项目反馈采集

```
触发条件：项目交付后
执行人：你主动记录

操作步骤：
[你] "XX项目验收，客户反馈是XXX"
[AI] 1. 解析反馈内容
     2. 提取关键词和情绪倾向
     3. 生成改进建议
     4. 写入Elite L2记忆（layer=feedback, type=client_project）
```

### 3.3 反馈分类体系

每条反馈记录必须携带以下分类标签：

```
一级分类（来源）：
  ├─ 公众号
  ├─ LoRA训练
  ├─ 客户项目
  └─ 内部运营

二级分类（信号性质）：
  ├─ 正信号（效果优于预期）
  ├─ 负信号（效果低于预期）
  └─ 中性信号（持平或数据不足）

三级分类（影响域）：
  ├─ 标题/选题
  ├─ 内容质量
  ├─ 训练参数
  ├─ 数据质量
  ├─ 客户满意度
  └─ 流程效率
```

## 第四章：记忆层 — Elite记忆系统操作

### 4.1 记忆写入规范

#### 4.1.1 写入时机清单

| 触发事件 | 写入层级 | 内容 | 优先级 |
|---------|---------|------|-------|
| 公众号文章发布后 | L2 + WAL | 文章元数据+效果反馈 | P0 |
| LoRA训练完成 | L2 + L6 | 训练参数+效果评分 | P0 |
| 感知层信号采集 | L2 + L4 | 外部信号摘要 | P1 |
| SOP更新 | L4 + L5 | 流程变更记录 | P1 |
| 标准变更 | L5 + L6 | 验收标准更新 | P1 |
| 客户反馈 | L2 + L3 | 客户原始反馈 | P1 |
| 日常运营数据 | L2 | 指标快照 | P2 |

#### 4.1.2 写入模板

**L2向量库写入（语义搜索用）**：
```
elite.py remember "<内容>" -c "<分类标签>" --layer L2
```

**L6知识图谱写入（关系检索用）**：
```
elite_knowledge.py add-entity --name "<实体名>" --type "<实体类型>" --layer_source L2
elite_knowledge.py add-relation --subject "<主体>" --predicate "<关系>" --object "<客体>"
```

#### 4.1.3 写入示例

```
# 公众号文章效果写入
elite.py remember "Day 166: 一个公众号IP白名单，藏着2026年最大的隐喻" \
  -c "layer=feedback,type=公众号,style=KIWI日记,reads=800,shares=30,comments=5" \
  --layer L2

# 训练参数写入知识图谱
elite_knowledge.py add-entity --name "黑色LOVE睡衣_L2训练" --type "TrainingRun" \
  --description "试点SKU第一轮L2人物三视图训练，loss=0.15，综合评分4.2" \
  --attributes '{"date":"2026-06-17","sku":"黑色LOVE睡衣","layer":"L2","loss":0.15,"score":4.2}'
```

### 4.2 记忆检索规范

#### 4.2.1 检索时机清单

| 使用场景 | 检索内容 | 检索命令 |
|---------|---------|---------|
| 公众号选题 | 近期同类文章效果对比 | `elite.py search "公众号 阅读 分享 近期" --layer L2` |
| LoRA训练参数 | 同类SKU的历史最优参数 | `elite.py search "LoRA 训练 参数 SKU" --layer L2` |

| 知识图谱 | 实体关联关系查询 | `elite_knowledge.py traverse --name "黑色LOVE睡衣"` |

#### 4.2.2 跨层搜索策略

```
优先顺序：
1. L2向量库（语义匹配，快速定位）
2. L6知识图谱（关系查询，深度关联）
3. L4文件记忆（原始文档全文检索）
4. L3冷存储（历史归档数据）

搜索策略：
- 关键词精确搜索 → L4 + L3
- 语义模糊搜索 → L2
- 实体关系查询 → L6
- 时间范围查询 → L1 register
```

## 第五章：SOP层 — 标准化流程体系

### 5.1 SOP文件规范

每条SOP必须包含以下字段：

```
编号: SOP-XXX
名称: [流程名称]
版本: vX.X
最后更新: YYYY-MM-DD
负责人: [人/AI]
触发条件: [什么情况下启动]
输入: [需要什么]
输出: [产出什么]
执行步骤:
  1. [步骤一] — [执行人] — [预计时间]
  2. [步骤二] — [执行人] — [预计时间]
  ...
校验标准: [怎么算完成]
异常处理: [出问题怎么办]
```

### 5.2 核心SOP清单

#### SOP-001：公众号内容生产闭环

**触发条件**：新素材到达 / 定时触发 / 手动触发
**输入**：素材/选题/数据反馈
**输出**：公众号草稿箱就绪的文章

**执行步骤**：
```
1. 触发信号接收 — AI — <1min
2. 素材收集（WebSearch+Elite检索+对话记录） — AI — 3min
3. 选题决策（比对三条判断规则+已有数据） — AI → 人确认 — 2min
4. 内容生产（写作Agent → 视觉Agent → 格式Agent） — AI — 5min
5. 审核校验（26条checklist逐条过） — AI — 2min
6. 公众号HTML格式化 — AI — 1min
7. 推送到草稿箱 — AI — <1min（需要access_token有效）
8. 你登录公众号后台 → 确认 → 发布 — 人 — 2min
```

**校验标准**：
- 审核清单26条全部通过（不含N/A项）
- 不踩封禁清单5条任何一条
- 公众号排版预览正常（无CSS变量失败）

#### SOP-002：LoRA三层生图任务执行

**触发条件**：新SKU启动
**输入**：商品正反图 / 人物参考图 / SKU参数
**输出**：标注好的训练数据集

**执行步骤**：
```
P1: 确认SKU参数（款号/颜色/套装结构） — 人 — <5min
P2: L1商品正反图采集（正面/背面/套装分拍×2-4张） — 人/拍摄 — 依素材而定
P3: 确定人物档案（3-5人，trigger word+特征描述） — 人 — 10min
P4: L2人物三视图生成（每人正/侧/背，trigger word分离） — 人/生图工具 — 依素材而定
P5: L3多场景扩展（按姿态/场景/色板分批迭代） — 人/生图工具 — 分批进行
P6: Captioning（40-100词/张，人物特征和衣服特征分开） — AI辅助 — 30min/SKU
P7: 训练数据验收（去重/质量检查/词频分析） — AI → 人确认 — 10min
```

**校验标准**：
- 每张图都有一个独立的、清晰标注的caption
- 同一人物的所有图使用同一trigger word
- 不同人物使用不同的trigger word

#### SOP-003：反馈数据采集与写入

**触发条件**：公众号发布24小时后 / LoRA训练完成后 / 客户项目交付后
**输入**：原始数据（截图/日志/你主动报告）
**输出**：Elite记忆中的反馈记录

**执行步骤**：
```

## 第七章：正反馈机制搭建

### 7.1 正反馈的定义

正反馈 = **优化指标提升 → 驱动更好的决策 → 进一步优化指标**

### 7.2 正反馈回路设计

#### 回路1：公众号内容回路

```
文章发布 → 阅读量数据 → 标题分析 → 标题公式优化
                                     ↓
                  下一篇标题效果更好 → 提升阅读量
```

**实现方式**：
```
记录字段：标题结构 | 阅读量 | 对比均值
优化动作：Elite L2记录标题公式变化
驱动逻辑：好的标题公式被保留并复用，差的被淘汰
```

#### 回路2：LoRA训练回路

```
训练完成 → 五维评分 → 参数对比 → 参数空间优化
                                     ↓
                下一个SKU训练用更优参数 → 效果更好
```

**实现方式**：
```
记录字段：训练参数 | 测试评分 | 对比最优值
优化动作：Elite L6图谱中标记最优参数组合
驱动逻辑：同品类/同类型SKU训练自动参考最优参数
```

#### 回路3：客户服务回路

```
项目交付 → 客户反馈 → 问题归类 → SOP优化
                                     ↓
                下一个同类项目用更优SOP → 客户更满意
```

### 7.3 正反馈监控指标

| 指标 | 当前基准值 | 优化目标 | 监控频率 |
|------|-----------|---------|---------|
| 公众号平均阅读量 | 650 | 1000 | 每周 |
| 文章完成率 | 63% | 75% | 每周 |
| LoRA训练平均评分 | 4.2 | 4.5 | 每轮训练 |
| 审核清单通过率 | 98% | 100% | 每次 |
| SOP更新周期 | 30天 | 15天（更快迭代） | 每月 |

### 7.4 负反馈管理机制

**负反馈不等于失败，负反馈是系统的养料。**

**管理流程**：
```
1. 采集负反馈信号（训练失败/文章表现差/客户投诉）

```

```
elite.py remember "L1正反图拍摄遮挡导致L2 ControlNet参考错误" \

## 第十一章：故障处理手册

### 11.1 故障分级

| 级别 | 定义 | 响应时限 | 负责人 |
|------|------|---------|--------|
| P0 | 系统不可用，数据可能丢失 | 立即 | 人工介入 |
| P1 | 核心功能异常，但不丢数据 | 24小时内 | AI排查→人工确认 |
| P2 | 非核心功能异常 | 72小时内 | AI处理即可 |
| P3 | 体验问题/小Bug | 下个迭代 | 记录，定期修复 |

### 11.2 常见故障处理流程

#### 故障1：Elite记忆检索返回空结果

```
P2级别故障

可能原因：
1. L2向量库未写入数据
2. 语义搜索关键词偏差
3. LanceDB索引损坏

排查步骤：
1. elite.py status → 检查L2数据量
2. elite_warmstore.py status → 检查向量库状态
3. 尝试L4+L3层搜索兜底 → elite_coldstore.py search --keyword "<关键词>"

修复：
1. 如果是索引损坏 → 重新build索引
2. 如果是无数据 → 检查写入日志（elite_wal.py stats）
3. 如果确认数据丢失 → 从L3恢复
```

#### 故障2：公众号草稿箱推送失败

```
P1级别故障

可能原因：
1. access_token过期
2. IP白名单变更
3. 草稿箱API限频

排查步骤：
1. 重新获取access_token验证 → curl api.weixin.qq.com/cgi-bin/token
2. 检查IP是否在白名单中
3. 检查API返回的错误码

修复（按顺序）：
1. 重新获取access_token（2小时有效）
2. 如果IP变更 → 你去公众号后台更新白名单
3. 如果限频 → 等待15分钟后重试
```

#### 故障3：反馈数据未写入Elite

```
P2级别故障

可能原因：
1. Elite记忆系统未运行
2. 反馈数据格式不符合规范
3. WAL日志写入异常

排查步骤：
1. elite_wal.py stats → 检查是否有PENDING或PROCESSING状态记录
2. elite_wal.py recover → 执行崩溃恢复

修复：
1. 执行崩溃恢复
2. 手动补录反馈数据
3. 检查数据格式后重试
```

#### 故障4：知识图谱实体重复

```
P3级别故障

排查步骤：
1. elite_knowledge.py list-entities --type "<类型>" → 列出所有实体
2. 检查是否有重复实体

修复：
elite_knowledge.py merge --name1 "<实体A>" --name2 "<实体B>"
```

### 11.3 数据恢复操作

#### 从L3冷存储恢复数据到L2

```
elite_coldstore.py search --keyword "<关键词>"          # 在L3中搜索
elite_coldstore.py restore --id <记录ID>                 # 恢复到L2
elite_warmstore.py add --content "<恢复内容>" -c "<分类>" # 写入L2向量库
```

#### 从快照恢复全系统

```
1. elite_coldstore.py snapshot --restore <快照文件路径>
2. elite_wal.py recover
3. elite.py status → 验证恢复结果
```
