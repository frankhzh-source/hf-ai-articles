# 我给 AI 造了一个跨设备、永不丢失的记忆系统：Elite 六层架构实践

每次打开新的 AI 对话，它都像失忆一样从头问你是谁、在做什么。更麻烦的是，我在笔记本上沉淀的偏好和结论，换到台式机上就全没了。AI 越用越聪明的前提，是它得记得住。

过去一年我试过把记忆塞进 prompt、塞进笔记、塞进各种"长期记忆"插件，最后都不够用：要么检索不准，要么跨设备不同步，要么一次崩溃全丢。所以我搭了一套自己的方案——**Elite 记忆系统**，六层架构，飞书 + GitHub 双云后端，两台电脑实时一致。

这篇文章讲清楚它怎么设计的，以及踩过的坑。

## 一、为什么是六层，而不是一个文件

记忆不是"存下来"就完事。不同信息有不同生命周期：

- 当前对话的临时状态，关掉就该清空；
- 三个月前的一次会议结论，需要随时能搜到；
- 你反复强调的"千万别用模板痕迹"，值得被永久钉在策展层。

把这些都丢进同一个文件，检索会崩、维护会乱。Elite 用六层把"热度"和"寿命"分开：

| 层 | 角色 | 存储介质 |
|----|------|----------|
| L1 Hot RAM | 当前会话状态 | SESSION-STATE.md |
| L2 Warm | 语义向量检索 | LanceDB 本地向量库 |
| L3 Cold | 冷存储归档 | 本地文件 |
| L4 Curated | 人工策展的精华 | MEMORY.md |
| L5 Cloud | 跨设备云同步 | 飞书 + GitHub |
| L6 Auto | 从日记自动抽取候选记忆 | 本地启发式 |

还有一条 **WAL（Write-Ahead Log）** 横切在所有写操作之前：任何记忆落地前先写日志，崩溃了能重放，保证不丢。

## 二、L2 语义检索：本地中文模型，不联网也能搜

最关键的是"搜得准"。Elite 用 BAAI 的 `bge-small-zh-v1.5` 中文 embedding 模型，512 维向量，跑在 LanceDB 本地向量库里。

几个工程决策值得说：

- **本地推理，不依赖云端 API**。模型约 130MB，首次下载后完全离线，隐私和稳定性都更好。
- **中文友好**。bge 系列对中文语义对齐做了专项优化，比通用多语言模型在中文记忆检索上准很多。
- **LRU 热度跟踪**。每次搜到的片段 access_count 自动 +1，越常用的越容易被召回，等于一个轻量的"记忆强化"机制。

`ingest` 一条命令就能把用户级笔记、项目日记批量向量化；`search "飞书文档和腾讯文档区别"` 直接返回带来源和层级的片段。

## 三、L5 双云后端：飞书管日常，GitHub 管镜像

跨设备是这套系统真正的价值。L5 同时挂两个后端：

- **飞书云空间**：主同步通道。`lark-cli` 把记忆镜像推到指定文件夹，手机、网页、另一台电脑都能直接看。
- **GitHub 私有仓库**：仓库级镜像。代码和记忆文本都在里面，等于一份带版本历史的备份。

一条 `cloud push` 会**同时**推飞书和 GitHub；一条 `cloud pull --apply` 会把云端记忆合并回本地，再 `ingest` 重建向量索引。

### 一个真实的工程坑：git 443 被网络重置

GitHub 原走 git 协议推送，但我的办公网络会把 `github.com:443` 的 git 推送连接重置。解决办法是让 `push` 改走 **`gh api` 的 Contents API**（走 `api.github.com`，gh 已认证），逐文件比对 sha、内容没变就跳过、变了就 PUT。单文件 ≤1MB，记忆文本完全够用，顺带绕开了网络限制。

另一个坑是**双后端冲突**：飞书 pull 会把最新记忆写进本地已跟踪的 `cloud_sync/` 目录，导致 git 工作树"脏"，挡住后续 git pull。修复方式是 pull 前只 `stash` 这个目录、绝不碰机器专属的 `config.json`，恢复冲突时以云端权威版本为准。

## 四、每日自动化：两台电脑最终一致

光有同步命令不够，人不会每天手动跑。我建了一个**每日自动化**：

`cloud pull --apply → ingest → cloud push`

每天自动把另一台机器的更新拉下来合并、重建索引、再推回。遇到冲突或失败会通知我。

两台机器**同时改同一个文件**时，后推的一方可能遇到远端 sha 过期（GitHub 返回 422），这时不静默丢数据，而是告警并等次日自动化收敛——异步最终一致，比强行覆盖安全。

## 五、常用命令速查

```
python cli.py status              # 看六层状态
python cli.py ingest              # 重建向量索引
python cli.py search "关键词"      # 语义检索
python cli.py cloud push          # 推飞书 + GitHub
python cli.py cloud pull --apply  # 拉取并合并
python cli.py wal verify          # 检查防丢日志是否全部同步
```

## 结语

AI 记忆系统的本质，是把"你反复告诉我的事"变成**可检索、可跨设备、可恢复**的资产。六层架构不是为了炫技，是把"临时状态 / 语义检索 / 冷归档 / 人工策展 / 云同步 / 自动抽取"各自放对位置。

如果你也在搭自己的 AI 记忆层，飞书 + GitHub 双后端 + 每日自动化这条组合，我实测半年，稳。
